## 核心数据库设计规范

### 1️⃣ 命名统一：小写 + 下划线

* 表名：复数名词，如：users, orders, conversation_logs
* 字段名：user_id, created_at, is_deleted
* 不要用 CamelCase：❌ UserInfo, createdAt
* 👉 口诀：“表复数，小写蛇（snake_case）。”
### 2️⃣ 必须有主键 id，用 BIGINT

* 每张表必须有：id BIGINT（或 BIGSERIAL）
* 不用业务字段当主键（比如邮箱、用户名都不要当主键）
### 3️⃣ 每张表都要有时间字段

* 最少这两个：
* created_at TIMESTAMP
* updated_at TIMESTAMP

👉 口诀：“有表就有时间线。”
### 4️⃣ 尽量用软删除，不要直接删

* 加两个字段：
* is_deleted BOOLEAN DEFAULT FALSE
* deleted_at TIMESTAMP NULL

真正删除很少用，一般只在日志、临时表里。
👉 口诀：“删记录先打标，不直接删。”
### 5️⃣ 外键怎么用？

* 小项目 / 一般后台 / SaaS / AI Demo：✅ 可以用外键（方便保证数据不乱）
* 高并发 / 将来要分库分表时： 去掉外键约束，留 xxx_id 字段，用业务代码保证
* 你现在可以记：
* 👉 “现在可以用外键，将来做大再去掉。”
### 6️⃣ 索引：只建在“经常查”的字段上

* 一定要给：主键、user_id、created_at 这类常用查询字段建索引
* 不要给：gender、is_deleted 这种区分度极低的字段单独建索引
* 多个条件常一起查 → 组合索引
* 👉 口诀：“常查建索引，少查别乱建。”
### 7️⃣ 大字段单独放一张“详情表”

* 例子：
* articles：id、title、author_id、created_at
* article_contents：id、article_id、content（TEXT）
* 👉 原则：核心表瘦一点，大内容丢详情表。
### 8️⃣ 字段类型要“对得上”

* 时间：用 TIMESTAMP / TIMESTAMPTZ，不要 VARCHAR
* 金额：用 NUMERIC(12,2)，不要 FLOAT，更不要 VARCHAR
* 布尔：用 BOOLEAN，不要 INT(1) + “0/1 约定”
* 👉 口诀：“钱用 NUMERIC，时间别用字符串。”
### 9️⃣ 状态用枚举字符串，不用写死数字

* 例子：
* status VARCHAR(20) -- 'pending' / 'paid' / 'failed'
* 不要：
* status INT -- 0,1,2 然后自己记含义，半年后你自己都看不懂

👉 口诀：“状态用英文词，不要神秘数字。”
### 🔟 日志 / 聊天记录这类表要有心理预期：会很大

* 暂时不用分表没关系，但要记住：
* 这类表只追加不更新
* 一定有 user_id + created_at
* 后面如果变大，要分库分表，就按这两个字段拆
* 👉 口诀：“大表先规整，将来好分家。”
